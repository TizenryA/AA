# Tavo《织梦大陆·种田RPG》正则踩坑记录（v1）

## TL;DR（先看这个）
- **先让它稳定渲染，再谈美化**：一开始用“一个萝卜一个坑”的极简替换，别上来就复杂循环/花式作用域。
- **标签名必须统一**：输出里写了 `<资产>`，正则就必须匹配 `<资产>`，不要混用 `<asset>`。
- **避免正则引擎兼容雷区**：不要用 `(?s)`/`(?m)` 这类内联 flag；多行一律用 `([\s\S]*?)`。
- **捕获组引用用 `$1/$2`**：别用 `${1}` 这类写法（Tavo 侧可能不认）。
- **决策栏别用“循环替换”赌引擎**：最稳的是“一次性抓 A/B/C/D 四段 → 直接生成 2×2 按钮”。
- **严格对齐「落英标准卡」字段风格**：`placement:[2,1]`、`markdownOnly:true`、`runOnEdit:false`、`trimStrings:[]`。
- **绝大多数脚本都用“显示时”**：避免接收/改写导致开场白失效、原始记录丢失；压缩脚本默认关闭，单独设“发送/发出时”。

---

## 1) 标签不一致导致整块不渲染
**现象**：面板里“资产”块没渲染，或某条 alternate_greetings 面板直接错位。

**原因**：输出用了 `<asset>...</asset>`，但正则只匹配 `<资产>...</资产>`（或反过来）。

**解决**：
- 统一标签：固定使用计划书标签体系：`<农场><日期><天气><资产><作物><niko><任务><决策><地图></农场>`。
- 在卡文件里把所有历史遗留的 `<asset>` 批量替换成 `<资产>`。

**建议**：
- **写死约束**：在 system_prompt 里明确“不要生成其它相似标签”。

---

## 2) `(?s)`/`(?m)` 内联 flag 在 Tavo 侧不稳定
**现象**：正则列表能看到，但 UI 完全不出现；所有标签都像没匹配到一样。

**原因**：Tavo 的正则引擎对 `(?s)`/`(?m)` 这类内联 flag 兼容性不稳定。

**解决**：
- 多行匹配统一改成：`([\s\S]*?)`，并用非贪婪 `*?`。

**建议**：
- **一律不用内联 flag**，把兼容性风险降到最低。

---

## 3) 捕获组引用：`$1` vs `${1}`
**现象**：决策按钮只替换了一部分，或替换文本出现奇怪的残留。

**原因**：替换字符串里用了 `${1}` 这种语法，Tavo 侧不一定支持。

**解决**：
- 引用捕获组只用 `$1/$2/$3...`。

---

## 4) 决策栏循环替换不稳定（最常见大坑）
**现象**：
- A/B/C/D 只渲染出一两个按钮；
- 顺序错乱；
- 正文里出现误伤（被当成选项替换）。

**原因**：
- “先把 `<决策>` 包一层，再循环替换 `[A-D]`”这种方案依赖引擎重复执行/作用域，Tavo 上容易出现次数、顺序、范围不确定。

**解决（推荐）**：
- **一次性解析**：用一个正则从 `<决策>` 中抓出 A/B/C/D 四段文本，再直接生成 2×2。
- 同时兼容多种选项格式：`[A]`、`【A】`、`A.`、`A、`、`(A)`。

**建议**：
- 决策是“玩法入口”，必须走最稳方案，不在这里炫技。

---

## 5) “作用时机/placement 映射”一定要以标准卡为准
**现象**：你在卡里改了 placement，但在 Tavo 导入后看到的“显示/接收/改写/发出”不按预期。

**原因**：不同平台/版本对 v3 的 `placement` 映射不一致；我们一开始是按经验猜，导致走弯路。

**解决**：
- 以你提供的 **《Tavo_落英_标准输出格式.json》** 为准：
  - 渲染类脚本：`placement: [2,1]`
  - 标准卡里还出现另一组：`placement: [1,2]`（用于特殊用途，如隐藏某段）。

**建议**：
- 所有 UI 渲染脚本都统一到 `[2,1]`，先跑通。
- “压缩/剥离面板”脚本默认 **disabled=true**，且单独放到“发出/发送”通道（具体 placement 以你的 Tavo 显示为准）。

---

## 6) 落英标准字段组合（强制对齐）
我们最终确认：对齐标准卡能显著减少“导入后行为怪异”。

**推荐字段最小集合**（渲染类脚本）：
- `placement: [2,1]`
- `markdownOnly: true`
- `runOnEdit: false`
- `trimStrings: []`
- `promptOnly: false`
- `substituteRegex: 0`

---

## 7) 禁用“接收/改写”以避免原始数据丢失
**现象**：开场白无法执行/历史消息被永久改写，原始 `<农场>...</农场>` 消失。

**原因**：把正则放在接收/改写时机会改写聊天记录。

**解决**：
- UI 渲染全部走“显示时”。
- 发送压缩脚本单独走“发出/发送”，且默认关闭。

---

## 8) 调试方法：先做“探针”再细化
**做法**：
- 加一个最早执行的脚本，在消息最前面插入 `【正则已执行】`。
- 再插入一个简单 HTML 方块作为“HTML 渲染探针”。

**意义**：能快速二分定位：
- 正则有没有执行？
- HTML 渲染有没有被前端禁止？

---

## 9) 最终推荐：极简稳定模板（可复制）
> 思路：一个标签一个脚本，全部显示时；多行用 `[\s\S]*?`；决策一次性抓 A/B/C/D。

- 开标签（CSS/JS 注入、容器开启）：匹配 `<农场>`
- 各区块：
  - `<日期>...` → `<div class="...">...`
  - `<天气>...` → ...
  - `<资产>...` → ...
  - `<作物>...` → ...
  - `<niko>...` → ...
  - `<任务>...` → ...
  - `<地图>...` → ...
- 决策：一次性解析 A/B/C/D → 2×2
- 闭合：匹配 `</农场>`
- 压缩脚本：`<农场>[\s\S]*?</农场>` → 空字符串（disabled=true，发出/发送时）

---

## 10) 本次项目的“正确工作顺序”（以后照这个走）
1. 先定输出标签体系（计划书）
2. 用极简正则跑通渲染（一个萝卜一个坑）
3. 决策栏用一次性解析做稳定
4. 再做 Tabs/剧情窗口等游戏化
5. 最后才考虑发送压缩与性能优化
